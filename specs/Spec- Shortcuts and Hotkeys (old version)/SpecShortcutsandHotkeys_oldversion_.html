Spec: Shortcuts and Hotkeys        *** DRAFT ***

Introduction:

A desktop application would not be complete without shortcuts, mostly menu shortcuts. There is no difference with a web application, Montage needs to provides support for shortcuts.

At the time this specs is written, Montage does not offer a menu component, however shortcut could be used by an application without menus.

There is two types of key sequences: shortcuts and hotkeys. A shortcut consists of multiple keys pressed at the same time, usually including at least one modifier key such as command, alt, control, option or shift (i.e: CMD+Z, CTRL+X, CMD+SHIFT+TAB). A Hotkey consists of only one key (i.e: TAB, ENTER, ESC).

In the context of Montage, there is no reason to differentiate hotkeys from shortcuts, therefore when this document uses the term shortcuts, it means shortcuts and hotkeys, unless say otherwise.

Due to the complex disparity in handling key events across browsers, platform and keyboard layout (l10n), shortcuts needs to stay simple in order to be compatible as much as possible, ideally shortcut should uses only letters A to Z, numbers 0 to 9 and some punctuation. Even so, it would be almost impossible to guarantee that a specific shortcut would work everywhere.

Another limitation is that a shortcut can only consists of modifiers and one other key. Montage will not support shortcut such as CMD+P+R. Hotkeys by definition are only one key.

 

Montage should provide an easy way to define shortcuts, either via serialization or directly from javascript. Ideally we should refrain from loading code related to shortcuts handling unless needed by the application.

Mac versus the rest of the world:

Macintosh uses the Command key (⌘) while the rest of the world (including iOS) uses the control key for standard shortcuts. ⌘+Z on Mac should be Control+Z on Windows and Unix.

The ⌘ key on Mac is in fact the META modifier key in DOM.

By using the term CMD or COMMAND in a shortcut, we will interpret it internally as META on Mac and CTRL on other system or devices:

        CMD+Z means META+Z on Mac and CTRL-Z on other systems

However, if you specify CTRL or META in the shortcut sequence, it will be interpreted as is on all platforms.

Shortcuts Event versus DOM Events:

When a shortcut is detected, we should dispatch a custom event. The sequence of evenement is:

  1. the user presses one or several keys
  2. DOM dispatches keydown, and potentially keypress events
  3. if we detect a specific sequence, we dispatch a shortcut event.
  4. the user release the keys
  5. DOM dispatches one or more keyup events
  6. If a shortcut was detected as the keydown or keypress phase, a shortcut event must be dispatched.

Sound straightforward, however there is two main difference between a regular DOM event and a shortcut event:

  1. A DOM events is targeted to the active element while a shortcut event might not!
  2. If a shortcut event is stopped, any related keydown and keypress, or keyup events must be also stopped (same for prevent default).

Rejected Solution 1: Shortcut Manager:

The Shortcut Manager in a singleton which is instantiated by Montage Core on startup. It adds two new methods to the Object.prototype:

Object.addShortcutListener(keys, name, listener, capture)

Object.removeShortcutListener(keys, name, listener, capture)

shortcut is a string that describe the key(s) to be pressed: “command+Z”

name is the name of the shortcut. We need to use a name as the shortcut itself could be localized in order to support international keyboard layout.

Listener and capture as identical to Object.addEventListener and Object.removeListener.

When calling Object.addShortcutListener and Object.removeShortcutListener, the Shortcut Manager will take care of adding or removing an event listener on the object as well updating the manager’s internal shortcut map used for a quick detection of shortcuts.

The Shortcut Manager also defines a serializer/deserializer unit to allow the user to define shortcuts directly in the template. Here is an example of serialization:

            "undoButton": {

               "module": "montage/ui/button.reel",

               "name": "Button",

               "properties": {

                   "element": {"#": "undoButton" },

                   "identifier": "undo",

                   "preventFocus": true,

                          “shortcut”: {name:”undo”, keys:”cmd+Z”}

               },

               "listeners": [

                   {

                       "type": "action",

                       "listener": {"@": "delegate"},

                       "capture": false

                   }

               ],

                    "shortcuts": [

                   {

                       "keys": "cmd+Z",

                                "name": "undo",

                       "listener": {"@": "delegate"},

                       "capture": false,

                                “global”: true

                   }

               ],

           }

The shortcut manager listens for keydown and keypress event at the document level. When it detect a shortcut, it dispatches a custom event named shortcut<Name>.

In the example above, the delegate would need to define a method called handleShortcutUndo(event).

PROS:

  1. Simple and intuitive serialization
  2. Simple and intuitive APIs

CONS:

  1. The shortcut manager is always loaded. This could be optimized by only loading a shortcut proxy with minimum code and load the remaining code asynchronously when needed
  2. How to address the targeting issue of the shortcut in serialization in a clean way
  3. Do we really need a shortcut block? Why now just a shortcut property on controls

Rejected Solution 2: Shortcut Composer

The shortcut composer is a composer Object that is instantiated by each components that care for shortcuts. Each shortcut composer will listen for keyboard events and when it detects a shortcut, will dispatch a shortcut event to the component it’s bounded with.

Here is an example of serialization:

           "shortcutComposer": {

               "prototype": "montage/ui/composer/shortcut-composer",

               "properties": {

                   "component": {"@": "undoButton"},

                   "shortcuts": [

                       {

                           "shortcut": "cmd+Z"

                           "name": "undo"

                       }

                   ]

               },

               "listeners": [

                   {

                       "type": "shortcut",

                       "listener": {"@": "delegate"}

                   }

               ]

           },

"undoButton": {

               "module": "montage/ui/button.reel",

               "name": "Button",

               "properties": {

                   "element": {"#": "undoButton" },

                   "identifier": "undo",

                   "preventFocus": true

               },

               "listeners": [

                   {

                       "type": "action",

                       "listener": {"@": "delegate"},

                       "capture": false

                   }

               ]

           }

The shortcut composer listens for keydown and keypress event at the composer’s component or element level. When it detect a shortcut, it dispatches a custom event named shortcut<Name>.

In the example above, the delegate would need to define a method called handleUndoShortcut(event).

PROS:

  1. Code only loaded when needed
  2. No need to register shortcuts, just set ShortcutComposer.shortcuts
  3. We could specify if the composer should listen at the component level of document level (maybe by not specifying a component and element, the composer will then listen on the document instead)

CONS:

  1. Required to serialize an extra object
  2. Serialization less intuitive
  3. Multiple composers doing the same work on key events to detect is a shortcut has been typed

Proposed Solution: The Virtual Keyboard

In order to solve the targeting issue of the shortcut manager solution as well the multiple instances of the composer solution and more,  a different approach have been taken with the notion of a virtual keyboard and virtual keys. Basically a consumer will register a virtual key with the virtual keyboard and then listen for event on that specific virtual key.

The virtual keyboard is a singleton (as long you use the defaultVirtualKeyboard) and virtual keys are shared as long they are representing the same key sequence and have the same identifier.

This solution address the potential conflict of two consumers registering the same key with a different identifier or two keys with the same identifier but different keys.

Creating a VirtualKeyboard:

The best way to create a virtual keyboard is to get the defaultKeyboard, however you can still create your own keyboard if needed. Here is 3 different way to instantiate a virtual keyboard object:

  1. var keyboard = require("montage/core/event/virtualKeyboard").defaultVirtualKeyboard;
  2. var VirtualKeyboard = require("montage/core/event/virtualKeyboard").VirtualKeyboard;
    var keyboard = VirtualKeyboard.create();
  3. “keyboard”: { "object": "montage/core/event/virtualKeyboard[defaultKeyboard]" }

Registering a virtual key and listening for event:

Once you have a virtual keyboard, you can register virtual keys. Here is how to add a Undo shortcut:

var undoVK = keyboard.registerKeySequence("command+z", "undo")

Now, you can start listening for one of the two virtual key events virtualKeydown and virtualKeyup using the method addVirtualKeyListener:

[object].addVirtualKeyListener(undoVK, this, false, false)

Object could be your component, an HTML element, document or window. If you wish to receive virtual key events what ever you have focus or not, just listen on the document or window.

Note: While virtual key event are similar to DOM event and are handled the same way, you must use the specific addVirtualKeyListener to register the listener.

VirtualKeyboard API:

(VirtualKeyboard) VirtualKeyboard[defaultKeyboard]

return the default virtual keyboard object

(VirtualKey) VirtualKeyboard.registerKeySequence(keys, identifier)

Arguments:        

string keys

the key sequence to register

string identifier

the identifier for the virtual key

Register a specified keys sequence and return a VirtualKey object.

VirtualKeyboard.unregisterKeySequence(keys, identifier)

Arguments:

string keys

the key sequence to unregister

string identifier

the identifier for the virtual key

Unregister the specified keys sequence from the virtual keyboard. You could also use VirtualKey.unregister() to unregister a virtual key.

VirtualKey Api:

To create a virtual key, you must call one of the virtual keyboard method or setter.

string VirtualKey.keys

The key sequences represented by the virtual key. The value returned by VirtualKey.keys is normalize and therefore might differ from the original value.

string VirtualKey.identifier

The virtual key identifier

VirtualKeyboard VirtualKey.keyboard

The keyboard to whom the virtual key belong to

VirtualKey.unregister()

Unregister the virtualKey and remove its event listener

Note: The VirtualKey's properties are immutable. Once set, you cannot modify them.

VirtualKey Events:

A virtual key will generate two custom events when the user press the specific sequence registered with the virtual key: virtualKeydown and virtualKeyup.

You can listen to either one and this in the capture or bubble phase. If you stop the event, the corresponding native key event will be also stopped (keydown or keyup). If you stop a virtualKeydown, the corresponding keypress event will also be stopped.

The detail property of the virtualKeydown and virtualKeyup custom event is the virtualKey object itself.

Use the special  [aComponent|application].addVirtualKeyListener and [aComponent|application].removeVirtualKeyListener to install or remove a listener:

[aComponent|application].addVirtualKeyListener(virtualKey, listener, onKeyup, useCapture)

[aComponent|application].removeVirtualKeyListener(virtualKey, listener, onKeyup, useCapture)

Arguments:

Object virtualKey

the VirtualKey object to listen to

Object|function listener

the listener

boolean onKeyup

Whatever to listen on virtualKeyup or virtualKeydown

boolean onKeyup

Whatever to be call during the capture or bubble phase

VirtualKey Event Handlers:

virtualKeydown and virtualKeyup are Montage custom event and therefore can be handled using the following syntaxes:

handleEvent(event)

captureVirtualKey[down|up](event)

handleVirtualKey[down|up](event)

capture[identifier]VirtualKey[down|up](event)

handle[identifier]VirtualKey[down|up](event)

Control's shortcut property:

The VirtualKeyboard and VirtualKeys are a generic way to define shortcut and hotkeys. However in most case this is too verbose and complex.

To simplify that, controls should provide a shortcut property to simulate their default action like clicking on a button or selecting a menu item  If a control has more than one action, it can provide several shortcut properties. Such shortcuts should only be enabled when the control is the active element.

Adding shortcut properties to Montage's controls should probably be the subject or another document.
